# 코드 반영 사항 — 인프라(cgv-infra) 작업 중 발견

> **관계**: 인프라 엔지니어가 코드를 k3s 환경에 배포하다, 코드에 고쳐야 할 게 드러나 **개발자 repo(cgv-onprem)에 올리는 PR**들. 인프라·코드가 별도 repo라 여기 모아 추적. 코드 변경 자체는 cgv-onprem에서 수행.
> **읽는 법**: 각 항목 = 차트가 뭘 하려는지 → 문제 → 뭐에 대한 건지 → 필요한 것 → 앱쪽 수정. (인프라가 백엔드에게 설명하듯.)
> 상태 = 완료 / 대기 / 진행.

| # | 항목 | 상태 |
|---|---|---|
| 1 | frontend → nginx-unprivileged | 완료 |
| 2 | booking 마이그레이션 도구(Flyway) 도입 | 대기 |
| 3 | Redis Sentinel HA — 클라이언트 sentinel化 | 보류 (dev=standalone 배포, HA는 prd 코드변경) |
| 4 | queue-go 토픽 자동생성 RF=1 하드코딩 제거 | 대기 |
| 5 | booking application.yml redis password 필드 | 대기 (prd redis auth 시) |

> ★ 정합검증(2026-07) 반영: dev Redis는 **standalone + auth off**로 배포 가능(#3 Sentinel HA는 prd 업그레이드로 보류). #4 = GitOps가 선언한 RF3을 queue 토픽 자동생성(RF=1 하드코딩)이 무력화할 위험 → 브로커 기본값(default.replication.factor=3)에 맡기게. #5 = prd에서 redis auth 켤 때 `spring.data.redis.password: ${REDIS_PASSWORD:}` 필요.

## 1. frontend → nginx-unprivileged (완료)

**차트가 하려는 것.** `charts/app`은 모든 앱을 쿠버네티스 **PodSecurity "restricted"**로 배포한다 — 컨테이너를 비루트(runAsNonRoot)·읽기전용 루트FS·권한상승 금지로 띄워, 컨테이너가 뚫려도 피해를 최소화하는 프로덕션 표준.

**문제.** frontend 이미지가 `nginx:1.27-alpine`인데, 이건 **root로 실행 + 80 포트(1024 미만=특권 포트)**를 연다. restricted PSA가 파드를 거부해서 frontend이 안 뜬다.

**뭐에 대한 건가.** "특권 포트(<1024)를 열려면 root여야 한다"는 리눅스 규칙. 비루트 컨테이너는 1024+ 포트만 열 수 있다. 그래서 비루트 nginx는 8080을 쓴다.

**필요한 것.** 비루트로 실행되고 8080을 듣는 nginx 이미지.

**앱쪽 수정 (cgv-onprem/frontend).**
- `Dockerfile`: `nginx:1.27-alpine` → `nginxinc/nginx-unprivileged:1.27-alpine`(uid 101), EXPOSE 8080.
- `nginx.conf`: `listen 80` → `listen 8080`. (proxy location은 유지 — 로컬 compose 게이트웨이용, k3s는 Ingress가 /api를 먼저 가로채 이 proxy를 안 탐.)
- `docker-compose.yml`: `8088:80` → `8088:8080`.
- 확인: `docker compose up --build`로 8088 정상.

## 2. booking 마이그레이션 도구(Flyway) 도입 (대기)

**차트가 하려는 것.** 앱을 여러 replica로 띄우려 한다(HA·부하분산). 이때 DB 스키마는 **딱 한 번만** 만들어져야 한다. 그래서 차트는 표준으로 **`DDL_AUTO=none` + PreSync 마이그레이션 Job**(앱 파드들이 뜨기 전에 스키마를 1회 적용)을 둔다.

**문제.** booking `pom.xml`에 **Flyway/Liquibase가 없다.** 스키마 생성 수단이 Hibernate `ddl-auto`뿐인데, 이건 각 파드가 기동하며 스키마를 만드는 방식이라 **멀티 replica면 여러 파드가 동시에 `CREATE TABLE` 레이스**를 낸다. → 지금은 어쩔 수 없이 **dev를 단일 replica + `DDL_AUTO=update`**로 우회 중(로컬 compose와 동일).

**뭐에 대한 건가.** **선언적 스키마 마이그레이션.** 스키마를 버전관리된 SQL(`V1__init.sql`, `V2__...`)로 두고 배포 때 1회 적용 → 멱등·재현·롤포워드 가능. `ddl-auto=update`는 개발 편의용이지 프로덕션/멀티팟용이 아니다(순서·삭제·정합 보장 없음).

**필요한 것.** booking에 마이그레이션 도구 + 초기 스키마 SQL. Job이 앱보다 먼저 `flyway migrate` 1회.

**앱쪽 수정 (cgv-onprem/booking).**
- `pom.xml`: `flyway-core` + `flyway-mysql` 의존.
- `src/main/resources/db/migration/V1__init.sql`: 현재 엔티티(Movie·Screening·Seat·Booking·BookingSeat) 스키마. **멱등 예매용 UNIQUE 제약** 검토(예: (screening_id, seat) 중복 예매 차단).
- `application.yml`: `spring.flyway.enabled=true`, `ddl-auto`는 그대로 env(`none`).
- 이후 차트: `migration.enabled=true`(PreSync Job이 `flyway migrate`) + `DDL_AUTO=none` + replica 증설.

## 3. Redis Sentinel HA — 클라이언트 sentinel化 (진행, B 결정)

**차트가 하려는 것.** `charts/redis` 래퍼가 bitnami redis를 **Sentinel HA**(master1 + replica2 + Sentinel3)로 배포한다. master가 죽으면 **Sentinel이 replica 하나를 새 master로 자동 승격**한다.

**문제.** 그 승격이 앱에 도달하려면 클라이언트가 **"지금 master가 누구냐"를 Sentinel에게 물어 재접속**해야 한다. 그런데 —
- queue: `goredis.NewClient(단일 Addr)` — 고정 주소 하나만 본다(`NewFailoverClient` 아님).
- booking: `spring.data.redis.host`(단일), sentinel 설정 없음.
→ master가 바뀌어도 둘 다 **죽은 옛 master 주소를 계속 물어** = failover가 앱까지 안 온다. **작동하지 않는 HA.**

**뭐에 대한 건가.** **Redis Sentinel** = master 감시·자동 승격·클라에게 새 master 통지하는 3종 세트. 핵심은 마지막 — 클라가 **sentinel-aware**여야(Sentinel 주소로 접속해 master를 조회) failover가 앱에 반영된다. 아니면 Sentinel은 뒤에서 승격만 하고 앱은 못 따라간다.

**필요한 것.** sentinel-aware 클라이언트(둘 다).

**앱쪽 수정 (PR 대상).**
- **queue (cgv-onprem/queue-go)** — `redis/client.go`의 `goredis.NewClient` → **`goredis.NewFailoverClient`**:
  ```go
  goredis.NewFailoverClient(&goredis.FailoverOptions{
      MasterName:    cfg.RedisMasterName,   // = 차트 sentinel.masterSet (cgvmaster)
      SentinelAddrs: cfg.RedisSentinelAddrs, // ["redis:26379"] 등
      Password:      password,
      // ReadTimeout/WriteTimeout 500ms 등 기존 옵션 유지
  })
  ```
  - `config.go`에 `REDIS_MASTER_NAME`, `REDIS_SENTINEL_ADDRS` env 추가(기존 `REDIS_HOST` 대체 또는 병행).
- **booking (cgv-onprem/booking)** — `application.yml`의 `spring.data.redis.host` 대신 **sentinel**:
  ```yaml
  spring:
    data:
      redis:
        sentinel:
          master: ${REDIS_MASTER_NAME:cgvmaster}
          nodes: ${REDIS_SENTINEL_NODES:redis:26379}
  ```
- **정합(렌더로 확정)**: masterName = `cgvmaster`. 앱 접속 = **Service `redis`의 sentinel 포트 `26379`**(6379 아님) → data 네임스페이스면 `REDIS_SENTINEL_NODES=redis.data.svc.cluster.local:26379`, `REDIS_MASTER_NAME=cgvmaster`.
- **연쇄**: 코드가 sentinel化되면 queue/booking의 env 값도 `REDIS_HOST`(단일) → 위 sentinel env로 교체(envs/dev/values-queue.yaml·values-booking.yaml).
