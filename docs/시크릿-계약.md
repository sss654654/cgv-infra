# SealedSecret 계약 (빌드 체크리스트)

> 클러스터별로 kubeseal로 만들어야 할 Secret 목록 + **정확한 키명**. 차트마다 키명이 달라 통일 불가 — 이 표대로 만들어야 붙는다.
> 절차: sealed-secrets 컨트롤러 up(install.sh [5]) → 각 Secret을 `kubeseal`로 봉인 → 해당 네임스페이스에 SealedSecret 매니페스트 배치.
> ⚠️ MinIO 3종(loki/mimir/tempo)은 **같은 MinIO 크레덴셜**을 각 차트가 요구하는 **다른 키명**으로 담는다(또는 컴포넌트별 MinIO 유저 발급).

| Secret 이름 | 네임스페이스 | 키(정확히) | 쓰는 곳 | 비고 |
|---|---|---|---|---|
| `minio-root-secret` | observability | `rootUser`, `rootPassword` | MinIO 루트 | rootPassword 8자+ |
| `loki-s3-credentials` | observability | `AWS_ACCESS_KEY_ID`, `AWS_SECRET_ACCESS_KEY` | Loki S3 | AWS SDK 기본 체인 |
| `mimir-minio-credentials` | observability | `MIMIR_S3_ACCESS_KEY_ID`, `MIMIR_S3_SECRET_ACCESS_KEY` | Mimir S3 | expand-env |
| `tempo-s3-credentials` | observability | `S3_ACCESS_KEY`, `S3_SECRET_KEY` | Tempo S3 | expand-env |
| `grafana-admin` | observability | `admin-user`, `admin-password` | Grafana 관리자 | |
| `mysql-secret` | data | `mysql-root-password` | MySQL 서버 루트 | ★ booking-secrets MYSQL_PASSWORD와 **동일값** |
| `redis-secret` | data | `redis-password` | Redis auth (prd만) | dev=standalone auth off → **불필요** |
| `booking-secrets` | app | `MYSQL_PASSWORD` | booking DB 접속 | = mysql-secret 값 (+prd redis auth면 `REDIS_PASSWORD`) |

**조건부:**
- queue가 Redis auth를 쓰면(현재 dev는 미설정) queue에도 `REDIS_PASSWORD` envFrom 시크릿 필요.
- MinIO S3 접근 = 루트 크레덴셜 재사용 가능하나, 이상적으론 MinIO에 loki/mimir/tempo 전용 유저를 만들어 최소권한 부여(빌드 때 결정).

## 서비스 DNS 정합 확인 (배포 후)
조사에서 일부 게이트웨이 서비스명이 버전/릴리스명 의존 — 배포 후 실제 이름으로 아래를 맞춰야 함:
- **Mimir 게이트웨이**: **`mimir-gateway`로 확정**(정합검증에서 helm template 렌더로 확인. `mimir-nginx`는 오기였음). grafana 데이터소스·alloy remote_write가 이걸 가리킴.
- **Loki 게이트웨이**: `loki-gateway`(기본 활성). 끄면 `loki:3100`.
- **Tempo**: HTTP 쿼리 `tempo:3200`, OTLP `tempo:4317/4318`.
- **prometheus-operator-crds**: Alloy가 **ServiceMonitor·PodMonitor CRD 둘 다** 필요(alloy 차트가 안 깖) → install.sh [6]이 둘 다 제공해야 함.
