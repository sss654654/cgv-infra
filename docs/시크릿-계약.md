# SealedSecret 계약 (빌드 체크리스트)

> 클러스터별로 kubeseal로 만들어야 할 Secret 목록 + **정확한 키명**. 차트마다 키명이 달라 통일 불가 — 이 표대로 만들어야 붙는다.
> 절차: sealed-secrets 컨트롤러 up(install.sh [7]) → 각 Secret을 `kubeseal`로 봉인 → 해당 네임스페이스에 SealedSecret 매니페스트 배치.
> ⚠️ MinIO 3종(loki/mimir/tempo)은 **lgtm 전용 유저(minio-lgtm-user) 크레덴셜**을 각 차트가 요구하는 **다른 키명**으로 담는다(IAM 격리 — 루트 재사용 금지).

| Secret 이름 | 네임스페이스 | 키(정확히) | 쓰는 곳 | 비고 |
|---|---|---|---|---|
| `minio-root-secret` | observability | `rootUser`, `rootPassword` | MinIO 루트 | rootPassword 8자+ |
| `minio-lgtm-user` | observability | `password` | MinIO lgtm 유저(IAM 격리) | 3버킷 한정 정책 lgtm-rw. loki/mimir/tempo S3 크레덴셜의 출처 |
| `loki-s3-credentials` | observability | `AWS_ACCESS_KEY_ID`, `AWS_SECRET_ACCESS_KEY` | Loki S3 | AWS SDK 기본 체인 |
| `mimir-minio-credentials` | observability | `MIMIR_S3_ACCESS_KEY_ID`, `MIMIR_S3_SECRET_ACCESS_KEY` | Mimir S3 | expand-env |
| `tempo-s3-credentials` | observability | `S3_ACCESS_KEY`, `S3_SECRET_KEY` | Tempo S3 | expand-env |
| `grafana-admin` | observability | `admin-user`, `admin-password` | Grafana 관리자 | |
| `mysql-secret` | data | `mysql-root-password` | MySQL 서버 루트 | booking-secrets의 MYSQL_PASSWORD와 **동일값** |
| `redis-secret` | data | `redis-password` | Redis auth (서버, data ns) | **dev HA도 필수**(Sentinel auth on). 클라 비번은 app ns로 복제(아래 queue-secrets·booking-secrets) |
| `booking-secrets` | app | `MYSQL_PASSWORD`, `REDIS_PASSWORD` | booking DB·Redis 접속 | MYSQL_PASSWORD=mysql-secret 값 · REDIS_PASSWORD=redis-secret 값(dev HA) |
| `queue-secrets` | app | `REDIS_PASSWORD` | queue Redis 접속 | = redis-secret 값(dev HA). cgv-app은 envFrom만 지원→키명이 env명과 같아야 함 |

**조건부:**
- **dev HA = Redis auth on** → queue도 `REDIS_PASSWORD` 필요 → queue-secrets(app ns) envFrom(위 표 반영). redis-secret은 data ns(서버)라 app 파드가 직접 못 봄 → app ns로 같은 값 복제.
- **mysql-secret 배달**: GitOps(sealed-secrets App wave -2)로 배달 → mysql App(wave -1)이 그 뒤 sync. 수동 apply 게이트 없음. root-app 전 봉인·커밋 선행 필수.
- **MinIO IAM 격리**: 루트 재사용 금지. lgtm 전용 유저(minio-lgtm-user, 정책 lgtm-rw = 3버킷 한정 CRUD)를 만들어 loki/mimir/tempo에 부여. negative test(타 버킷·admin API 거부)는 배포 후 mc로 확인.

## 서비스 DNS 정합 확인 (배포 후)
조사에서 일부 게이트웨이 서비스명이 버전/릴리스명 의존 — 배포 후 실제 이름으로 아래를 맞춰야 함:
- **Mimir 게이트웨이**: **`mimir-gateway`로 확정**(정합검증에서 helm template 렌더로 확인. `mimir-nginx`는 오기였음). grafana 데이터소스·alloy remote_write가 이걸 가리킴.
- **Loki 게이트웨이**: `loki-gateway`(기본 활성). 끄면 `loki:3100`.
- **Tempo**: HTTP 쿼리 `tempo:3200`, OTLP `tempo:4317/4318`.
- **prometheus-operator-crds**: Alloy가 **ServiceMonitor·PodMonitor CRD 둘 다** 필요(alloy 차트가 안 깖) → install.sh [8]이 둘 다 제공해야 함.
